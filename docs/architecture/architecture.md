# 架构升级方案 —— 照片服务 + 数据备份 + 资源管理

## 一、你的新需求梳理

### 需求 A：个人网站增强
- 数据自动备份到本地服务器
- 后台可以快速上下架资源（文章、页面等）

### 需求 B：照片服务器
- **公开区**：个人摄影作品展示（美观、体验好）
  - 支持按地点/时间/类型筛选和检索
  - 照片可附带词条、属性
- **私人区**：朋友家人照片（不对外开放）
  - 密码/权限保护
  - 可快速上下架照片
- **资源备份**：照片自动备份

### 关键架构问题
1. 照片服务是独立项目还是整合到个人网站？
2. 云服务器能否只做反向代理，真正的服务跑在本地？
3. 2核2G 够不够？如何分配？

---

## 二、云服务器 vs 本地服务器 —— 本质区别

```
云服务器（ECS）                    本地服务器（你的电脑/NAS）
├── 公网 IP（7x24 可访问）         ├── 没有固定公网 IP
├── 带宽有限（通常 1-5Mbps）       ├── 带宽充足（局域网千兆）
├── 存储小且贵（通常 40-100G 磁盘）├── 存储大且便宜（几TB很常见）
├── 2核2G，性能有限               ├── 性能通常更强
└── 适合：转发请求、轻量服务       └── 适合：存储密集、计算密集
```

**你的直觉是对的** —— 用 ECS 做"网络入口"，本地做"真正的后端"，是非常经典的架构。

---

## 三、三种架构方案对比

### 方案 A：全部部署在 ECS

```
浏览器 ──→ ECS（Nginx + Nuxt + SQLite + 照片存储）
```

| 优点 | 缺点 |
|------|------|
| 架构最简单 | 2G 内存跑所有服务，紧张 |
| 所有东西在一个地方 | 照片存储空间很快就不够 |
| | ECS 带宽小，大图加载慢 |
| | 数据备份不方便 |

**结论：❌ 不推荐。** 2核2G 跑个人网站还行，加上照片服务会很吃力。

---

### 方案 B：ECS 反向代理 + 本地服务器（推荐 ⭐⭐⭐）

```
浏览器
   │
   ▼
┌─────────────────────────┐
│  ECS（2核2G）             │
│  Nginx 反向代理（~10MB）  │  ← 极轻量，只转发请求
│  ├── site.example.com    │──→ 本地服务器：个人网站
│  └── photo.example.com   │──→ 本地服务器：照片服务
└─────────────────────────┘
        │ (通过 frp 隧道)
        ▼
┌─────────────────────────────────────────┐
│  本地服务器（你的电脑/NAS）                │
│                                          │
│  ┌──────────────────┐  ┌──────────────┐ │
│  │  个人网站 Nuxt    │  │  照片服务     │ │
│  │  ├── 前台页面     │  │  ├── 公开相册 │ │
│  │  ├── 博客        │  │  ├── 私密相册 │ │
│  │  └── 后台管理     │  │  └── 后台管理 │ │
│  └──────────────────┘  └──────────────┘ │
│                                          │
│  ┌──────────────────────────────────────┐│
│  │  数据库 + 照片文件 + 自动备份         ││
│  └──────────────────────────────────────┘│
└──────────────────────────────────────────┘
```

| 优点 | 缺点 |
|------|------|
| ECS 压力极小（只跑 Nginx） | 本地服务器需要保持开机 |
| 照片存储空间几乎无限 | 需要配置 frp 隧道（一次配好就行） |
| 本地带宽大，照片加载快 | 本地断网时网站不可访问 |
| 数据全在本地，备份方便 | |
| 节省 ECS 费用（不用买大硬盘） | |

---

### 方案 C：混合部署（最佳平衡 ⭐⭐⭐⭐）

```
浏览器
   │
   ▼
┌──────────────────────────────────────────┐
│  ECS（2核2G）                              │
│  Nginx 反向代理                            │
│  ├── /            → 个人网站（Nuxt 轻量版）│  ← 博客、自我介绍
│  ├── /admin       → 后台管理              │
│  ├── /api/backup  → 备份同步 API          │
│  └── /photo/*     → frp → 本地照片服务    │
└──────────────────────────────────────────┘
        │ (frp 隧道，只转发照片相关请求)
        ▼
┌──────────────────────────────────────────┐
│  本地服务器                                │
│  ┌────────────────────────────────────┐  │
│  │  照片服务（独立 Docker 容器）         │  │
│  │  ├── 公开相册（摄影作品）            │  │
│  │  ├── 私密相册（家人朋友，密码保护）   │  │
│  │  ├── 照片元数据（地点/时间/标签）     │  │
│  │  └── rsync 自动备份 → 外置硬盘      │  │
│  └────────────────────────────────────┘  │
└──────────────────────────────────────────┘
```

| 优点 | 缺点 |
|------|------|
| 个人网站在 ECS，始终可访问 | 架构稍复杂 |
| 照片在本地，存储大、速度快 | 需要学 Docker + frp |
| 照片服务挂了不影响主站 | |
| 数据分布合理，各自备份 | |

**结论：⭐ 推荐方案 C。**

理由：
1. 个人网站（博客+介绍）很轻量，2核2G 完全够
2. 照片服务是存储密集型，放本地更合理
3. 照片服务挂了，主站不受影响
4. 本地服务器可以随时开机/关机控制资源上下线

## 四、核心技术点说明

### 4.1 什么是 frp？（让本地服务器"有"公网 IP）

```
你家的电脑没有公网 IP，外面访问不了。

frp 的作用：
1. 本地电脑主动连到 ECS（建立一条"隧道"）
2. 当有人访问 ECS 的某个端口时
3. frp 把请求通过隧道转发到你的本地电脑
4. 本地电脑处理完，结果原路返回

相当于：ECS 是"前台接待"，本地电脑是"幕后工作者"
```

frp 是国内最流行的内网穿透工具，开源免费，配置简单。

### 4.2 资源上下架实现思路

```vue
<!-- 数据库中每个资源都有一个 status 字段 -->
<!-- admin 后台点击"下架"，就是把 status 改为 hidden -->
<!-- 前台查询时只显示 status=published 的内容 -->

<!-- 数据库设计 -->
Resource {
  id:          自增ID
  title:       标题
  type:        类型（article/photo/video）
  status:      状态（published/draft/hidden）
  is_public:   是否公开（true/false）
  tags:        标签（JSON）
  location:    拍摄地点
  taken_at:    拍摄时间
  created_at:  创建时间
}
```

### 4.3 自动备份方案

```
┌─────────────┐    rsync 定时同步    ┌──────────────┐
│  ECS 数据    │ ──────────────────→ │  本地备份目录  │
│  (网站+DB)   │    每天凌晨3点       │  /backup/ecs/ │
└─────────────┘                      └──────────────┘

┌─────────────┐    rsync 定时同步    ┌──────────────┐
│  本地照片    │ ──────────────────→ │  外置硬盘     │
│  /photos/    │    每天凌晨4点       │  /backup/photo│
└─────────────┘                      └──────────────┘

或者用 Docker Volume + 定时任务自动备份 SQLite
```

## 五、照片服务方案选择

照片管理有两条路：

### 选项 1：自建（整合到 Nuxt 项目）

用 Nuxt 3 写自定义照片模块：
- 公开画廊 + 私密相册
- 自定义标签、搜索、筛选
- 完全控制 UI 和功能
- **适合**：想学全栈开发、需求比较定制化

### 选项 2：用现成工具 + 集成

| 工具 | 特点 | 适合 |
|------|------|------|
| **Immich** | Google Photos 替代品，UI 极美，AI 人脸识别 | 功能齐全，开箱即用 |
| **PhotoPrism** | AI 标签、地图浏览、搜索强大 | 摄影作品管理 |
| **Lychee** | 极简相册，专注展示 | 纯展示需求 |

- 可以把 Immich 作为私人相册 + PhotoPrism 作为公开作品展示
- 通过 Nginx 统一入口，和主站风格整合

### 我的建议

```
公开摄影作品  →  自建 Nuxt 模块（定制化强，和主站风格统一）
私人相册      →  Immich（开箱即用，人脸识别好用，密码保护现成）
```

这样公开部分完全由你控制设计，私人部分用成熟方案省时省力。
