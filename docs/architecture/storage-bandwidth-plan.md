# 存储规划与带宽优化方案

## 一、带宽问题分析

你的担忧完全正确：

```
普通情况（照片在 ECS）：
用户 ←── 1份带宽 ──→ ECS（照片直接返回）

frp 转发情况（照片在本地）：
用户 ←── 1份带宽 ──→ ECS ←── 1份带宽 ──→ 本地服务器
                      ↑ 这里确实用了双倍带宽
```

ECS 公网带宽通常只有 1-5Mbps，双倍消耗会导致：
- 大图加载极慢
- 多人同时看照片直接卡死
- 流量费也可能增加

**结论：不能让所有照片都走 frp 转发。**

---

## 二、分层存储策略（核心方案）

把照片分为"热数据"和"冷数据"：

```
┌─────────────────────────────────────────────────────┐
│                    ECS（30GB）                        │
│                                                      │
│  ┌──────────────┐  ┌──────────────┐  ┌────────────┐ │
│  │  热数据层     │  │  应用层       │  │  系统层     │ │
│  │  近期照片     │  │  Nuxt 主站    │  │  OS + 工具  │ │
│  │  缩略图缓存   │  │  Nginx       │  │  Docker    │ │
│  │  （直接返回）  │  │  frp         │  │  日志       │ │
│  └──────────────┘  └──────────────┘  └────────────┘ │
│         ↑                                           │
│    用户直接访问，单倍带宽，速度最快                     │
└─────────────────────────────────────────────────────┘
          │
          │ rsync 定时同步（每天凌晨）
          ▼
┌─────────────────────────────────────────────────────┐
│                 本地服务器（无限制）                    │
│                                                      │
│  ┌──────────────┐  ┌──────────────┐                  │
│  │  冷数据层     │  │  全量备份     │                  │
│  │  历史照片     │  │  ECS 全站备份 │                  │
│  │  原始大图     │  │  照片全量     │                  │
│  │  Immich 私密  │  │              │                  │
│  └──────────────┘  └──────────────┘                  │
│         ↑                                           │
│    仅 frp 转发，访问频率低，可接受                      │
└─────────────────────────────────────────────────────┘
```

### 工作原理

| 场景 | 数据来源 | 带宽消耗 | 速度 |
|------|---------|---------|------|
| 浏览近期作品（最近3个月） | ECS 本地 | 1x | ⚡ 极快 |
| 浏览历史作品 | frp → 本地 | 2x | 🐢 较慢（可接受） |
| 私人相册（Immich） | frp → 本地 | 2x | 🐢 较慢（少量人访问） |
| 个人网站/博客 | ECS 本地 | 1x | ⚡ 极快 |

**关键优化：缩略图全部放 ECS，原图按需加载。**

## 三、ECS 30GB 存储分配方案

```
总计：30GB
├── 系统层                        6.0 GB
│   ├── OS + 系统软件              3.0 GB
│   ├── Docker Engine              1.0 GB
│   ├── Swap 分区                  2.0 GB
│   └── （2核2G 加 2G Swap 防止 OOM）
│
├── 应用层                        5.0 GB
│   ├── Nuxt 主站（代码+依赖）     1.0 GB
│   ├── Nginx                     0.5 GB
│   ├── frp                       0.5 GB
│   ├── SQLite 数据库             0.5 GB
│   ├── Node.js 运行时            0.5 GB
│   └── 日志 + 临时文件           2.0 GB
│
├── 热数据层（近期照片缓存）       15.0 GB
│   ├── 公开作品缩略图（WebP）     3.0 GB  ← 约 3000-5000 张
│   ├── 公开作品近期原图           8.0 GB  ← 约 200-400 张高清
│   ├── 首页/博客配图             2.0 GB
│   └── 临时上传区                2.0 GB
│
└── 预留缓冲                      4.0 GB
    └── 紧急空间 + Docker 镜像膨胀
```

**分配原则：**
- 系统+应用约占 11GB（固定开销）
- 照片缓存区 15GB（动态，定期清理旧数据）
- 预留 4GB 安全空间

## 四、照片同步策略

### 4.1 自动同步脚本（rsync）

```bash
# 每天凌晨 2 点：ECS 新增的照片 → 本地全量备份
0 2 * * * rsync -avz --delete /data/photos/ backup@local:/backup/photos/

# 每天凌晨 3 点：本地全量照片 → ECS（仅同步近3个月的热数据）
0 3 * * * rsync -avz --max-age=90 /photos/ root@ecs:/data/photos/hot/

# 每周清理：删除 ECS 上超过 3 个月的原图（缩略图保留）
0 4 * * 0 find /data/photos/hot/ -name "*.jpg" -mtime +90 -delete
```

### 4.2 智能缓存机制

```
用户请求照片 ID=12345
     │
     ▼
Nginx 判断：ECS 有这张照片吗？
     │
     ├── 有（热数据） → 直接返回 → ⚡ 1x 带宽
     │
     └── 没有（冷数据）
          │
          ▼
     Nuxt API 通过 frp 从本地获取
          │
          ▼
     缓存到 ECS（下次直接返回）
          │
          ▼
     返回给用户 → 🐢 2x 带宽（仅首次）
```

这个缓存策略可以用 Nuxt 服务端中间件实现：

```typescript
// server/middleware/photo-cache.ts
// 照片请求 → 先查 ECS 本地 → 没有再从 frp 获取并缓存
```

## 五、图片优化（带宽杀手的终极方案）

照片文件是带宽最大的消耗者，必须优化：

### 5.1 格式优化

```
原始照片：IMG_20260704.jpg → 8MB（4000x3000 像素）

处理后：
├── 缩略图（网格展示）  → thumb.webp    → 50KB   → 宽 400px
├── 中图（详情页预览）  → medium.webp   → 200KB  → 宽 1200px
└── 原图（下载/放大）   → original.jpg  → 8MB    → 按需加载

带宽对比：
- 直接加载原图：8MB × 20张 = 160MB ❌
- 先加载缩略图：50KB × 20张 = 1MB ✅
- 点击后再加载原图：按需
```

### 5.2 技术实现

```bash
# 使用 sharp（Node.js 图片处理库）自动生成三种尺寸
# 安装：pnpm add sharp

# 上传照片时自动处理：
# 1. 生成 WebP 缩略图（400px 宽，质量 80）
# 2. 生成 WebP 中图（1200px 宽，质量 85）
# 3. 保留原始文件
```

### 5.3 加载策略

```vue
<!-- 前端：懒加载 + 渐进式加载 -->
<template>
  <!-- 画廊网格：只加载缩略图 -->
  <PhotoGrid :photos="photos" loading="lazy" />

  <!-- 点击后：先显示模糊缩略图，再加载高清 -->
  <PhotoDetail>
    <BlurThumbnail :src="photo.thumb" />  <!-- 立即显示 -->
    <FullImage :src="photo.medium" />     <!-- 异步加载 -->
  </PhotoDetail>
</template>
```

## 六、frp 隧道优化

### 6.1 只转发必要的请求

```toml
# frpc.toml（本地端配置）
# 只转发 /photo/original/* 和 /api/immich/*
# 缩略图和中图全部在 ECS 本地处理，不走 frp

[[proxies]]
name = "photo-original"
type = "http"
localPort = 8080
customDomains = ["photo.yourdomain.com"]
locations = ["/photo/original", "/api/immich"]

[[proxies]]
name = "immich"
type = "http"
localPort = 2283
customDomains = ["photo.yourdomain.com"]
locations = ["/api/immich"]
```

### 6.2 带宽占用估算

| 访问场景 | 走 frp？ | 占 ECS 带宽 |
|---------|----------|------------|
| 浏览博客 | ❌ | 极少 |
| 看公开画廊（缩略图） | ❌ | 少（WebP 50KB/张） |
| 点击看大图（近期） | ❌（ECS 缓存） | 中（200KB/张） |
| 点击看大图（历史） | ✅ | 高（200KB/张，双倍） |
| 看私密相册 | ✅ | 高（但人少） |
| 下载原图 | ✅ | 很高（8MB，但偶尔） |

**90% 的日常浏览都不走 frp，带宽压力很小。**

## 八、总结

| 优化手段 | 效果 |
|---------|------|
| 分层存储（热/冷数据） | 90% 浏览不走 frp |
| WebP 缩略图 | 单张 8MB → 50KB，缩小 160 倍 |
| 懒加载 | 首屏只加载可视区域 |
| 缓存策略 | 热门照片常驻 ECS |
| frp 精确路由 | 只转发必要请求 |

**即使 ECS 只有 1Mbps 带宽，优化后也能流畅展示画廊。**
