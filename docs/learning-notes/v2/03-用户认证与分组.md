## 📖 核心概念：认证 vs 授权

**类比：** 
- **认证（Authentication）** = 进公司刷卡，证明"你是谁"
- **授权（Authorization）** = 进门后，你有权限进哪些房间

### 认证流程

```
用户输入密码 → 服务器验证 → 生成 JWT Token → 返回给前端
     ↓
前端每次请求带 Token → 服务器解 Token → 知道"这是谁"
     ↓
中间件检查权限 → 允许/拒绝访问
```

### JWT 是什么？

```
JWT = JSON Web Token = 一个加密的字符串，里面存着用户信息

结构：xxxxx.yyyyy.zzzzz
      ↓       ↓       ↓
    Header  Payload  Signature
    (算法)   (数据)   (签名)

Payload 里存的是：
{
  "id": 1,
  "role": "admin",
  "iat": 1625000000,  // 签发时间
  "exp": 1625086400   // 过期时间
}
```

## 💻 代码解读

### 统一权限模块（permission.ts）

```typescript
// ─── 来源判定 ─────────────────────────────────────────────────────────────────

// 可信 IP 列表（CIDR 格式）
const TRUSTED_CIDRS = [
  '127.0.0.0/8',      // 本地回环
  '10.0.0.0/8',       // 内网 A 类
  '172.16.0.0/12',    // 内网 B 类
  '192.168.0.0/16',   // 内网 C 类
]

export function getAccessOrigin(event: H3Event): AccessOrigin {
  const ip = getRequestHeader(event, 'x-forwarded-for') || 
             getRequestHeader(event, 'x-real-ip') || 
             '127.0.0.1'
  
  // 1. 检查是否是本地可信 IP
  for (const cidr of TRUSTED_CIDRS) {
    if (isInCidr(ip, cidr)) {
      return 'local_trusted'
    }
  }
  
  // 2. 检查是否是已登录的 owner
  const user = event.context.authUser
  if (user?.role === 'admin') {
    return 'remote_owner'
  }
  
  // 3. 检查是否是已登录的朋友
  if (user?.role === 'friend') {
    return 'remote_user'
  }
  
  // 4. 未登录
  return 'public'
}
```

**为什么用 CIDR 而不是直接比较 IP？**
```
❌ 直接比较：if (ip === '192.168.1.100')
   问题：内网 IP 可能变化，需要写死每个 IP

✅ CIDR 匹配：if (isInCidr(ip, '192.168.0.0/16'))
   好处：整个 192.168.x.x 网段都算内网
   192.168.0.0/16 表示前 16 位是网络号，后 16 位是主机号
```

### 登录中间件增强

```typescript
// server/middleware/auth.ts
export default defineEventHandler(async (event) => {
  const path = getRequestURL(event).pathname

  // 只拦截需要认证的路由
  if (!path.startsWith('/api/admin') && !path.startsWith('/api/ai')) return

  const authHeader = getRequestHeader(event, 'Authorization')
  if (!authHeader || !authHeader.startsWith('Bearer ')) {
    throw createError({ statusCode: 401, message: 'Authentication required' })
  }

  const token = authHeader.slice(7)  // 去掉 "Bearer " 前缀

  try {
    const decoded = jwt.verify(token, getJwtSecret()) as { id: number; role: string }
    
    // ⭐ 新增：从数据库获取完整用户信息（检查 status）
    const dbUser = await prisma.user.findUnique({ where: { id: decoded.id } })
    
    if (!dbUser) {
      throw createError({ statusCode: 401, message: 'User not found' })
    }
    
    // ⭐ 新增：禁用用户检查
    if (dbUser.status === STATUS.DISABLED) {
      throw createError({ statusCode: 403, message: 'Account is disabled' })
    }

    // 注入到 event.context 供后续 handler 使用
    event.context.auth = { id: decoded.id, role: decoded.role }
    event.context.authUser = toAuthUser(dbUser)
  } catch (error: any) {
    if (error.statusCode) throw error
    throw createError({ statusCode: 401, message: 'Invalid or expired token' })
  }
})
```

**关键点：`event.context` 是什么？**
```
event.context 是 H3 框架提供的"请求级上下文对象"

请求进入 → 中间件往 context 里塞数据 → 后续 handler 取出来用

类比：快递员（中间件）检查包裹后，在包裹上贴标签（context），
      仓库管理员（handler）看标签决定放哪个货架
```

## 🔍 对比与选型

### 认证方案对比

| 方案 | 原理 | 优点 | 缺点 | 适合 |
|------|------|------|------|------|
| JWT | 无状态 Token | 不需要服务端存储、跨域友好 | 无法主动失效 | 前后端分离 |
| Session | 服务端存储 | 可主动失效、安全 | 需要共享存储 | 传统 Web |
| OAuth 2.0 | 第三方授权 | 社交登录、安全 | 配置复杂 | 需要社交登录 |
| Passkey | 无密码认证 | 最安全、用户体验好 | 兼容性问题 | 新项目 |

**为什么选 JWT：** 前后端分离架构，JWT 无需服务端存储 session，Nuxt 3 SSR 友好。

### 权限中间件设计对比

| 方式 | 示例 | 优点 | 缺点 |
|------|------|------|------|
| 全局中间件 | auth.ts 拦截所有 /api/admin | 统一管理 | 不够灵活 |
| 路由级守卫 | 每个 API 文件单独检查 | 灵活 | 重复代码多 |
| 装饰器模式 | @RequireAuth() @RequireAdmin() | 声明式、优雅 | Nuxt 不原生支持 |
| 组合函数 | requireAuth(event) | 灵活、可复用 | 需要手动调用 |

**本项目选择：全局中间件 + 组合函数**
```
全局中间件：拦截 /api/admin 和 /api/ai 路由，注入 authUser
组合函数：在具体 handler 中调用 requireAdmin() / requireLocalAccess()
好处：中间件负责"认证"（你是谁），组合函数负责"授权"（你能做什么）
```

### 密码存储方案对比

| 方案 | 安全性 | 性能 | 说明 |
|------|--------|------|------|
| 明文存储 | ❌ 极差 | 最快 | 绝对不能用 |
| MD5/SHA | ❌ 差 | 快 | 可被彩虹表破解 |
| bcrypt | ✅ 好 | 慢（故意的） | 业界标准 |
| argon2 | ✅ 最好 | 可调 | 新一代标准 |

**本项目用 bcryptjs：** 纯 JS 实现，无需原生编译，Docker 部署友好。
