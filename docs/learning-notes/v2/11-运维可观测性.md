## 📖 核心概念：可观测性

**类比：** 可观测性就像汽车仪表盘——速度表（性能）、油量表（资源）、故障灯（告警），让你随时知道系统状态。

### 可观测性三支柱

```
1. 日志（Logs）     — 发生了什么？
   [2026-07-04 15:30] user=admin action=login ip=192.168.1.100

2. 指标（Metrics）  — 系统状态如何？
   CPU: 45%, Memory: 60%, Disk: 70%

3. 追踪（Tracing）  — 请求经过了哪里？
   请求 → API → 数据库 → 响应（耗时 120ms）
```

**本项目实现了：** 日志（审计日志）+ 基础指标（存储状态）

## 💻 代码解读

### 审计日志模型

```prisma
model AuditLog {
  id        Int      @id @default(autoincrement())
  userId    Int?                  // 操作者（系统操作时为 null）
  action    String                // 操作类型
  target    String?               // 操作对象
  detail    String?               // 详细信息（JSON）
  ip        String?               // 操作者 IP
  createdAt DateTime @default(now())
  
  user      User?    @relation(fields: [userId], references: [id])
}
```

### 记录审计日志

```typescript
// server/utils/audit.ts

export async function createAuditLog(
  userId: number | null,
  action: string,
  target?: string | null,
  detail?: any,
  ip?: string,
): Promise<void> {
  await prisma.auditLog.create({
    data: {
      userId,
      action,
      target,
      detail: detail ? JSON.stringify(detail) : null,
      ip,
    }
  })
}

// 使用示例
await createAuditLog(
  user.id,
  'delete_photo',
  `photo:${photoId}`,
  { filename: photo.filename, visibility: photo.visibility },
  getRequestHeader(event, 'x-forwarded-for')
)
```

**关键点：`JSON.stringify` 存储详情**
```
为什么用 JSON 字符串而不是单独的字段？

方式 1：单独字段
  detail_action: 'delete'
  detail_filename: 'photo.jpg'
  detail_reason: '不合适'
  → 每种操作需要不同字段，schema 膨胀

方式 2：JSON 字符串（本项目选择）
  detail: '{"action":"delete","filename":"photo.jpg","reason":"不合适"}'
  → 灵活，任何操作都可以存不同结构
  → 缺点：不能直接查询 JSON 内部字段（SQLite 不支持 JSON 查询）

权衡：灵活性 > 查询能力（审计日志主要是查看，很少需要按详情查询）
```

### 数据库备份脚本

```bash
#!/bin/bash
# scripts/backup-database.sh

BACKUP_DIR="/mnt/data/backups/database"
DB_PATH="/mnt/data/personal-website/prisma/prod.db"
TIMESTAMP=$(date +%Y%m%d_%H%M%S)
BACKUP_FILE="$BACKUP_DIR/prod_${TIMESTAMP}.db"

# 创建备份目录
mkdir -p "$BACKUP_DIR"

# 复制数据库（SQLite 支持在线备份）
cp "$DB_PATH" "$BACKUP_FILE"

# 压缩
gzip "$BACKUP_FILE"

# 保留最近 30 天的备份
find "$BACKUP_DIR" -name "*.gz" -mtime +30 -delete

echo "✅ 数据库备份完成: ${BACKUP_FILE}.gz"
```

**关键点：SQLite 在线备份**
```
SQLite 支持在线备份（不需要停止服务）

方式 1：直接 cp（本项目选择）
  cp prod.db backup.db
  简单，但如果 cp 时有写操作，可能得到不一致的快照

方式 2：sqlite3 .backup 命令
  sqlite3 prod.db ".backup backup.db"
  更安全，保证一致性

方式 3：VACUUM INTO
  sqlite3 prod.db "VACUUM INTO 'backup.db'"
  最安全，但会锁数据库

本项目用 cp：个人网站写入量小，风险可接受
```

### 媒体备份脚本

```bash
#!/bin/bash
# scripts/backup-media.sh

BACKUP_DIR="/mnt/data/backups/media"
PHOTOS_DIR="/mnt/data/personal-website/photos"
TIMESTAMP=$(date +%Y%m%d_%H%M%S)

# 创建带时间戳的备份目录
BACKUP_PATH="$BACKUP_DIR/photos_${TIMESTAMP}"
mkdir -p "$BACKUP_PATH"

# rsync 增量备份
rsync -avz --delete \
  "$PHOTOS_DIR/" \
  "$BACKUP_PATH/"

# 计算备份大小
SIZE=$(du -sh "$BACKUP_PATH" | cut -f1)
echo "✅ 媒体备份完成: $BACKUP_PATH ($SIZE)"
```

**关键点：rsync 增量备份**
```
rsync 的增量备份原理：
1. 对比源目录和目标目录
2. 只复制新增/修改的文件
3. 删除目标端多余文件（--delete）

首次备份：全量复制（可能很慢）
后续备份：只复制变化的部分（很快）
```

## 🔍 对比与选型

### 日志方案对比

| 方案 | 存储 | 优点 | 缺点 | 适合 |
|------|------|------|------|------|
| 数据库 | SQLite/MySQL | 可查询、结构化 | 性能差 | 小项目 |
| 文件 | .log 文件 | 简单、性能好 | 不方便查询 | 系统日志 |
| ELK Stack | Elasticsearch | 功能最强 | 重量级 | 企业 |
| 云服务 | CloudWatch/日志服务 | 托管、省心 | 有成本 | 云原生 |

**本项目选择数据库：** 审计日志量小，复用现有 SQLite，支持按用户/操作/时间查询。

### 备份策略对比

| 策略 | 原理 | 优点 | 缺点 | 适合 |
|------|------|------|------|------|
| 全量备份 | 每次备份所有数据 | 简单、恢复快 | 耗时、占空间 | 数据量小 |
| 增量备份 | 只备份变化部分 | 省空间、快 | 恢复需要多份 | 数据量大 |
| 差异备份 | 备份与上次全量的差异 | 平衡 | 空间逐渐增大 | 中等数据 |
| 快照 | 文件系统快照 | 秒级、一致性 | 需要特定 FS | 企业 |

**本项目选择增量（rsync）：** 照片数据量可能很大，增量备份节省空间和时间。

### 监控告警方案对比

| 方案 | 优点 | 缺点 | 适合 |
|------|------|------|------|
| 脚本 + 邮件 | 简单 | 需要配置邮件 | 小项目 |
| Prometheus + Grafana | 功能强 | 重量级 | 中大型 |
| 云监控 | 托管 | 有成本 | 云原生 |
| 本项目：脚本 + 文档 | 最轻量 | 手动检查 | 个人项目 |
