## 📖 核心概念：API 网关

**类比：** 网关就像公司前台——所有访客都要先经过前台登记，前台检查你的身份和权限，然后才让你进去见人。

### 为什么需要网关？

```
没有网关：
  用户 → 直接访问 Hermes API → 可能滥用、没有限制、没有记录

有网关：
  用户 → 网关 → 检查身份 → 检查权限 → 检查速率 → 记录日志 → 转发给 Hermes
         ↓
       所有 AI 请求都经过网关，网关是唯一的入口
```

### 网关架构

```
┌─────────┐     ┌─────────────┐     ┌─────────────┐
│  用户    │────→│  AI 网关    │────→│  Hermes     │
│ (前端)   │←────│  /api/ai/*  │←────│  (AI 引擎)  │
└─────────┘     └──────┬──────┘     └─────────────┘
                       │
              ┌────────┼────────┐
              ↓        ↓        ↓
         权限检查  速率限制  审计日志
```

## 💻 代码解读

### 网关代理 API

```typescript
// server/api/ai/chat.post.ts
export default defineEventHandler(async (event) => {
  const user = event.context.authUser
  
  // 1. 权限检查
  if (!user || !user.aiAccess) {
    throw createError({ statusCode: 403, message: 'AI 功能未开启' })
  }

  // 2. 速率限制检查
  const allowed = await checkRateLimit(user.id, 'ai_chat')
  if (!allowed) {
    throw createError({ statusCode: 429, message: '请求太频繁，请稍后再试' })
  }

  // 3. 读取用户消息
  const body = await readBody(event)
  const { message, conversationId } = body

  // 4. 转发给 Hermes
  const response = await callHermesAPI({
    message,
    userId: user.id,
    conversationId,
    accessLevel: user.aiAccessLevel,
  })

  // 5. 记录审计日志
  await createAuditLog(user.id, 'ai_chat', null, {
    messageLength: message.length,
    responseLength: response.length,
    conversationId,
  })

  return { success: true, data: { response } }
})
```

**关键点：为什么要限制速率？**
```
没有速率限制：
  用户每秒发 100 个请求 → 服务器崩溃 → 所有人都不能用

有速率限制：
  用户每秒最多 10 个请求 → 超过的返回 429 → 服务器正常运行

类比：高速公路收费站
  没有收费站：所有车同时涌入 → 堵死
  有收费站：控制车流 → 顺畅通行
```

### 速率限制实现

```typescript
// server/utils/rate-limit.ts

// 简单的内存速率限制（生产环境应该用 Redis）
const rateLimitStore = new Map<string, { count: number; resetAt: number }>()

export async function checkRateLimit(
  userId: number,
  action: string,
  maxRequests = 10,
  windowMs = 60000, // 1 分钟
): Promise<boolean> {
  const key = `${userId}:${action}`
  const now = Date.now()
  
  const record = rateLimitStore.get(key)
  
  if (!record || now > record.resetAt) {
    // 新窗口，重置计数
    rateLimitStore.set(key, { count: 1, resetAt: now + windowMs })
    return true
  }
  
  if (record.count >= maxRequests) {
    // 超过限制
    return false
  }
  
  // 未超过，计数 +1
  record.count++
  return true
}
```

**关键点：`Map` vs `Object`**
```typescript
// 用 Map
const store = new Map<string, number>()
store.set('key', 123)
store.get('key')     // 123
store.has('key')     // true
store.delete('key')  // 删除
store.size           // 元素个数

// 用 Object
const store: Record<string, number> = {}
store['key'] = 123
store['key']         // 123
delete store['key']  // 删除
Object.keys(store).length  // 元素个数

为什么用 Map？
1. Map 的键可以是任意类型（Object 只能是字符串）
2. Map 有 .size 属性，Object 需要 Object.keys().length
3. Map 的遍历性能更好
4. Map 可以用 .has() 检查键是否存在
```

### Mock AI Provider

```typescript
// server/services/ai-provider-mock.ts
// 开发/测试用的 Mock AI，不调用真实 API

export async function callMockAI(message: string): Promise<string> {
  // 模拟延迟
  await new Promise(resolve => setTimeout(resolve, 500))
  
  // 返回固定回复
  return `收到你的消息："${message}"。这是一个 Mock 回复，用于测试。`
}
```

**为什么要 Mock？**
```
开发时不想每次都调用真实 AI API：
1. 真实 API 要花钱
2. 网络可能不稳定
3. 响应时间不确定

Mock 的好处：
1. 免费
2. 稳定（固定回复）
3. 快速（模拟延迟 500ms）
4. 可控（可以模拟成功/失败/超时）
```

## 🔍 对比与选型

### 速率限制实现对比

| 方式 | 存储 | 优点 | 缺点 | 适合 |
|------|------|------|------|------|
| 内存 Map | 进程内存 | 最简单 | 重启丢失、不支持多实例 | 开发/单实例 |
| Redis | Redis 服务 | 持久化、支持多实例 | 需要额外服务 | 生产环境 |
| SQLite | 数据库文件 | 持久化、无额外依赖 | 并发性能差 | 小项目 |
| 云服务 | API Gateway | 开箱即用 | 有成本 | 云原生 |

**本项目选择内存 Map：** 单实例部署，重启丢失可以接受，不需要引入 Redis。

### AI 网关方案对比

| 方案 | 优点 | 缺点 | 适合 |
|------|------|------|------|
| 自建网关 | 完全可控、可定制 | 开发成本 | 需要精细控制 |
| API Gateway 云服务 | 开箱即用 | 成本、灵活性低 | 云原生 |
| Nginx 限流 | 简单、高性能 | 功能有限 | 简单场景 |
| 直接调用 | 最简单 | 无保护 | 开发测试 |

**为什么自建网关：** 需要检查用户 AI 权限等级、记录审计日志、控制对话上下文，这些逻辑云服务做不到。

### 对话存储方案对比

| 方案 | 优点 | 缺点 | 适合 |
|------|------|------|------|
| 内存 | 最快 | 重启丢失 | 开发 |
| SQLite | 持久化、简单 | 并发差 | 小项目 |
| MongoDB | 灵活 schema | 需要额外服务 | 大量对话 |
| Vector DB | 支持语义搜索 | 复杂 | RAG 应用 |

**本项目选择 SQLite：** 对话量小，复用现有数据库，不需要额外依赖。
