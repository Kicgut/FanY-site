## 📖 核心概念：分层存储

**类比：** 就像衣柜分层——常穿的衣服放最外层（ECS 缩略图），换季的放里层（本地原图），不穿的放储藏室（移动硬盘）。

### 存储架构

```
ECS 服务器（轻量入口）          本地服务器（重存储）
┌─────────────────────┐        ┌─────────────────────┐
│ /opt/personal-website│        │ /mnt/data/photos    │
│   /thumbnails/      │        │   /public/          │ ← 原图
│     /public/        │ ←同步← │   /friends/         │ ← 原图
│       2026-07/      │        │   /private/         │ ← 原图
│         xxx_thumb.jpg│        │     2026-07/        │
└─────────────────────┘        └─────────────────────┘
   ~100KB/张                      ~5MB/张
```

### 为什么分目录？

```
❌ 不分目录：
  /uploads/photo1.jpg  (public)
  /uploads/photo2.jpg  (private)  ← 哪个是公开的？哪个是私人的？
  /uploads/photo3.jpg  (friends)
  
✅ 按 visibility 分目录：
  /photos/public/2026-07/photo1.jpg
  /photos/friends/2026-07/photo2.jpg
  /photos/private/2026-07/photo3.jpg  ← 一目了然，且可以用文件系统权限控制
```

## 💻 代码解读

### 存储路径解析

```typescript
// server/services/photo-storage.ts

const PHOTOS_ROOT = '/mnt/data/personal-website/photos'
const THUMBNAILS_ROOT = '/mnt/data/personal-website/thumbnails'

export function resolvePhotoStoragePath(
  photo: { visibility: string; originalPath: string | null; filename: string },
  variant: 'original' | 'thumb' | 'medium',
): string {
  const baseUuid = photo.filename.replace(/\.[^.]+$/, '')
  const ext = 'jpg' // 缩略图统一用 jpg

  // 从路径中提取 YYYY-MM
  const dateMatch = photo.originalPath?.match(/(\d{4}-\d{2})/)
  const dateFolder = dateMatch ? dateMatch[1] : 'unknown'

  if (variant === 'original') {
    // 原图：photos/{visibility}/{YYYY-MM}/{filename}
    return join(PHOTOS_ROOT, photo.visibility, dateFolder, photo.filename)
  } else {
    // 缩略图：thumbnails/{visibility}/{YYYY-MM}/{uuid}_{variant}.jpg
    const thumbName = `${baseUuid}_${variant}.${ext}`
    return join(THUMBNAILS_ROOT, photo.visibility, dateFolder, thumbName)
  }
}
```

**关键点：UUID 文件名**
```
为什么不用原始文件名？

❌ 原始文件名：IMG_20260704_153022.jpg
   问题：
   1. 可能重复（两张不同照片都叫 IMG_xxx）
   2. 中文文件名可能乱码
   3. 暴露拍摄时间等隐私信息

✅ UUID 文件名：a1b2c3d4-e5f6-7890-abcd-ef1234567890.jpg
   好处：
   1. 全球唯一，不会重复
   2. 纯 ASCII，不会乱码
   3. 不泄露任何信息
```

### Sharp 缩略图生成

```typescript
async function getSharp() {
  if (sharpAvailable === false) return null
  if (sharpModule) return sharpModule
  try {
    sharpModule = (await import('sharp')).default
    sharpAvailable = true
    return sharpModule
  } catch {
    sharpAvailable = false
    console.warn('[photo-storage] sharp not available, falling back to copy')
    return null
  }
}

export async function generateThumbnail(
  inputPath: string,
  outputPath: string,
  options: { width?: number; height?: number; quality?: number } = {},
): Promise<string> {
  const sharp = await getSharp()
  
  if (!sharp) {
    // sharp 不可用时，直接复制原图作为缩略图
    await copyFile(inputPath, outputPath)
    return outputPath
  }

  const { width = 400, height = 300, quality = 80 } = options
  
  await sharp(inputPath)
    .resize(width, height, { fit: 'inside', withoutEnlargement: true })
    .jpeg({ quality })
    .toFile(outputPath)
    
  return outputPath
}
```

**关键点：`fit: 'inside'` 是什么？**
```
原图 1920x1080，目标 400x300

fit: 'cover'   → 裁剪到 400x300，可能丢失边缘内容
fit: 'contain' → 缩放到 400x225，有白边
fit: 'inside'  → 缩放到 400x225，无白边，保持比例 ← 本项目选择
fit: 'fill'    → 拉伸到 400x300，变形
```

### rsync 同步脚本

```bash
#!/bin/bash
# scripts/sync-thumbnails.sh
# 同步缩略图到 ECS

ECS_HOST="root@120.26.231.150"
ECS_THUMB_DIR="/opt/personal-website/uploads/thumbnails"
LOCAL_THUMB_DIR="/mnt/data/personal-website/thumbnails"

# --dry-run: 先预览，不实际执行
# --delete: 删除 ECS 上本地没有的文件
# -avz: 归档模式 + 详细输出 + 压缩传输

rsync -avz --dry-run \
  -e "ssh -i ~/.ssh/id_ed25519" \
  "$LOCAL_THUMB_DIR/" \
  "$ECS_HOST:$ECS_THUMB_DIR/"

# 去掉 --dry-run 才会真正同步
```

**rsync 常用参数：**
```
-a  归档模式（保留权限、时间戳等）
-v  详细输出
-z  压缩传输
-n  dry-run，只预览不执行
--delete  删除目标端多余文件
--exclude 排除某些文件
```

## 🔍 对比与选型

### 图片处理库对比

| 库 | 语言 | 优点 | 缺点 | 适合 |
|----|------|------|------|------|
| Sharp | Node.js | 快（libvips）、功能全 | 需要原生编译 | Node.js 项目 |
| Jimp | Node.js | 纯 JS、无需编译 | 慢 | 简单处理 |
| ImageMagick | 系统工具 | 命令行、功能最强 | 需要安装 | 脚本/批处理 |
| Pillow | Python | Python 生态 | 需要 Python | Python 项目 |

**为什么选 Sharp：** Nuxt 3 项目，Sharp 是 Node.js 生态最快的图片处理库，基于 libvips（比 ImageMagick 快 4-8 倍）。

### 文件同步方案对比

| 方案 | 优点 | 缺点 | 适合 |
|------|------|------|------|
| rsync | 增量同步、压缩、SSH 加密 | 需要 SSH | 服务器间同步 |
| rclone | 支持云存储（S3、OSS） | 配置复杂 | 云存储 |
| scp | 简单 | 全量复制 | 一次性传输 |
| Git LFS | 版本控制 | 不适合大量二进制 | 代码仓库 |

**为什么选 rsync：** 只同步变化的文件，压缩传输，SSH 加密，是服务器间文件同步的标准工具。

### 存储介质对比

| 介质 | 容量 | 速度 | 成本 | 适合 |
|------|------|------|------|------|
| SSD | 中 | 最快 | 贵 | 热数据（ECS 系统盘） |
| HDD | 大 | 中等 | 便宜 | 温数据（本地原图） |
| 移动硬盘 | 超大 | 慢 | 最便宜 | 冷数据（永久备份） |
| 云 OSS | 无限 | 取决于带宽 | 按量付费 | 需要 CDN 加速 |

**本项目的三层存储：**
```
ECS SSD  → 公开缩略图（热数据，访问频繁）
本地 HDD → 所有原图 + 私密内容（温数据，偶尔访问）
移动硬盘 → 永久备份（冷数据，几乎不访问）
```
