## 📖 核心概念：项目上下文加载

**类比：** 就像你接手一个新公司，第一天不是直接干活，而是先读公司的规章制度、过往项目文档、和团队约定。

### 为什么第一步是"读文档"？

```
新手程序员 ❌：接到需求 → 直接写代码 → 踩坑 → 返工
老手程序员 ✅：接到需求 → 读文档 → 理解约束 → 写代码 → 一次到位
```

### 本项目的关键约束（AGENTS.md 精华）

```
1. 身份权限：admin / friend / viewer
2. 访问来源：local_trusted / remote_owner / remote_user / public  
3. 操作类型：view / upload / edit / publish / delete / execute

三者组合决定权限：
  admin + local_trusted + delete = ✅ 允许
  admin + remote_owner + delete = ❌ 拒绝（远程不能删）
  friend + remote_user + view(private) = ❌ 拒绝（朋友不能看私人内容）
```

## 💻 代码解读

### 权限常量定义（permission.ts）

```typescript
export const STATUS = { ACTIVE: 'active', DISABLED: 'disabled', PENDING: 'pending' } as const
// ↑ as const 的意思是"这是常量，TypeScript 会推断出字面量类型"
// 没有 as const，TYPE 会被推断为 string，有了就是 'active' | 'disabled' | 'pending'

export const ROLES = { ADMIN: 'admin', FRIEND: 'friend', VIEWER: 'viewer' } as const
// ↑ 用户角色：管理员、朋友、普通访客

export const AI_LEVELS = { NONE: 'none', CHAT: 'chat', LIMITED: 'limited', OWNER: 'owner' } as const
// ↑ AI 访问等级：无权限、聊天、受限、完全权限
```

### 为什么用 `as const`？

```typescript
// 没有 as const
const STATUS = { ACTIVE: 'active' }
// TypeScript 认为 STATUS.ACTIVE 的类型是 string（可以是任何字符串）

// 有 as const
const STATUS = { ACTIVE: 'active' } as const
// TypeScript 认为 STATUS.ACTIVE 的类型是 'active'（只能是这个值）
// 好处：写代码时有精确的自动补全和类型检查
```

## 🔍 对比与选型

### 项目规范加载方式对比

| 方式 | 优点 | 缺点 | 适合场景 |
|------|------|------|---------|
| AGENTS.md 文件 | 所有 Agent 自动读取、版本可控 | 需要手动维护 | 多 Agent 协作项目 |
| README.md | GitHub 自动展示 | Agent 不一定读 | 纯人类阅读 |
| .cursorrules | Cursor 专用 | 其他 Agent 不读 | 单工具项目 |
| 内联注释 | 就近说明 | 分散难维护 | 小项目 |

**为什么选 AGENTS.md：** 本项目用 Hermes + Codex + Claude Code 多个 Agent，AGENTS.md 是通用约定，所有 Agent 都会优先读取。

### 权限模型设计对比

| 模型 | 示例 | 优点 | 缺点 |
|------|------|------|------|
| RBAC（角色） | admin/editor/viewer | 简单直观 | 无法区分访问来源 |
| ABAC（属性） | 基于 IP + 时间 + 角色 | 灵活 | 复杂 |
| ACL（访问控制列表） | 每个资源单独配置 | 精确 | 维护成本高 |
| 本项目：RBAC + 来源 | role + origin + action | 平衡安全和简单 | 需要自定义中间件 |
