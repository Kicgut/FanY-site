# Git 与协作治理

## 提交原则

- 一个提交只表达一个可验证的变更意图；代码、测试和必要文档应同提交。
- 不提交密钥、生产数据库、上传原图、备份、日志、构建产物或个人环境文件。
- 不覆盖用户已有的未提交改动；发现无关脏文件时保留并说明。

## 分支与推送

- `master` 是当前稳定主线。推送前确认 remote、账户和 SSH key 对应目标仓库。
- 提交前至少检查：`git diff --check`、相关测试或类型检查、`git status --short`。
- 部署相关提交必须保留可识别的镜像标签或回滚路径，并遵循 `operations/` 手册。

## 协作边界

- 代码修改前先读取相关架构、设计与规范；文档规则见 `docs/AGENTS.md`。
- 修改 API、数据模型、权限、部署或存储边界时，同步更新相应当前文档。
- 已完成的 task、plan、实施记录和验收记录不继续维护为独立文件；吸收有效结论后删除。

## 忽略规则

`.gitignore` 必须覆盖 `.env*`（保留安全模板时例外）、数据库副本、上传目录、备份、日志、依赖目录和构建输出。需要可复现实例时，使用经过脱敏的 `fixtures/` 或 `examples/`。
